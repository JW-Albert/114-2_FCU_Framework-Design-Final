🎬 開場 ｜ 封面 + 動機

第 1 頁 · 封面

大家好，我們這組要報告的是「公司車輛管理系統」。

先說明一下重點：這份報告的核心不是功能有多少，而是我們如何用物件導向設計原則，加上十個 GoF 設計模式，把一個會不斷擴充的系統，設計得可維護、可測試、可替換。

第 2 頁 · 動機

為什麼要做這個系統？企業車輛常用紙本或 Excel 借用，有三個痛點：第一，時段衝突往往事後才發現；第二，申請到還車的審核軌跡不透明；而最關鍵的第三點是難以擴充——通知方式跟衝突規則全寫死了，改一處就動全身。

最後這一點，正是設計模式要解決的問題，也是整份報告的主軸。

01:15–03:15

🗺 系統全貌 ｜ 概觀 + 技術 + 架構

第 3 頁 · 系統概觀

系統核心是一條借車工作流程：員工送出申請，進入 PENDING；管理員核准或拒絕；核准後標記出車，進入 IN_USE；最後還車。

圍繞這條主線有六大模組——認證、車輛、借車、審核、保養、違規。角色分管理員、主管、員工三種，其中主管只能審核同部門員工的申請。

第 4 頁 · 技術選型

技術上是 Spring Boot 3 加 Vue 3 的全棧架構。後端 Java 21、Spring Security 配 JWT 做無狀態認證、JPA 加 Flyway 管資料庫；前端 Vue 3 配 TypeScript 跟 Pinia。部署用 GitHub Actions、PM2 跟 Cloudflare Tunnel。

第 5 頁 · 分層架構

架構分五層，重點在中間這條 Repository 介面層——它是依賴反轉的邊界。

上面的 Service 與 Domain 只依賴介面，下面的 Infrastructure 用 JPA Adapter 去實作它們。也就是說，底層的持久化細節在編譯期被反轉掉，上層完全不感知 JPA 的存在，因此能脫離資料庫獨立測試。

03:15–04:15

🧩 設計模式總覽 + 領域模型

第 6 頁 · 模式總覽

接下來是報告的重點。我們實作了十個 GoF 設計模式，涵蓋行為型、創建型、結構型三大類，其中四個是後期用來重構既有設計的。

每一個模式都對應一個真實的擴充痛點，不是為了用而用。

第 7 頁 · 領域模型

這張類別圖是整個領域模型的全貌，幾個模式同時交會在這裡：右邊 BorrowingRequest 持有狀態介面，是 State；左邊 RoleFactory 建立角色，是 Factory；最右邊的 Publisher 與 Observer 則是 Observer 模式。先有個印象，接下來逐一拆解。

04:15–05:45

⚙️ 行為型三模式 ｜ State · Observer · Strategy

第 8 頁 · State

第一個是 State。借車申請有五種狀態，每種只允許特定轉換。如果用 if-else 判斷狀態字串，Service 會充滿分支。

解法是讓 BorrowingRequest 當 Context，把每種狀態的轉換邏輯封裝在各自的 ConcreteState。非法轉換——比方對已還車的申請再核准——會拋例外回 HTTP 422。新增狀態只要加一個類別，不改 Context。

第 9 頁 · Observer

第二個是 Observer。狀態變更要通知人，但通知方式不該寫死在 Service 裡。

解法是 Service 只負責發事件，Spring 自動注入所有 Observer。最棒的是，要新增一個通知管道，只要實作介面再標記 @Component，完全不用改任何現有程式碼——這就是開放封閉原則。

第 10 頁 · Strategy

第三個是 Strategy。衝突判斷的演算法有很多種——嚴格重疊、寬鬆同日、加緩衝。寫死在 Service，換邏輯就得改 Service 本身。

解法是抽成 ConflictCheckStrategy 介面，透過 Spring 注入。換策略只要換一個 @Bean 設定，Service 完全不動。

05:45–06:45

🛠 其餘模式 ｜ Template Method · Factory · Adapter

第 11 頁 · Template Method

第四個是 Template Method。每個 Service 的 public 方法都以相同的權限檢查開頭，違反 DRY。

解法是抽象基類定義「先驗證、後執行」的骨架，子類只用 lambda 提供業務邏輯，權限跟稽核集中在一處。

第 12 頁 · Factory Method

第五個是 Factory Method。需要依字串建立 Role 物件，直接 new 會產生對具體類別的依賴。

改用 RoleFactory 集中建立，呼叫端只依賴介面。新增角色，例如後來加的主管，只要加一個類別跟一行 case。

第 13 頁 · Adapter

第六個是 Adapter。Domain 定義了領域介面，但持久化用 JPA，兩者介面不相容。

每個 Adapter 實作 Domain 介面，內部委派 JPA，透過雙向轉換橋接。Service 完全不感知 JPA，五個 Adapter 結構都相同。

06:45–08:15

🔧 四個重構模式 ｜ Builder · Chain · Decorator · Command

接下來這四個，是我們後期專門用來重構、消除痛點的模式。

第 14 頁 · Builder

第七個 Builder：從資料庫還原物件時，舊做法要重播 approve、startUse 這些有副作用的方法，會觸發通知、邏輯也脆弱。新做法用 Builder 的 restoreState 直接設定目標狀態，繞過所有副作用。

第 15 頁 · Chain of Responsibility

第八個 責任鏈：送審要做權限、車輛存在、時段衝突三步驗證，原本全擠在一個方法。改成三個獨立 Validator，用 @Order 排序，Service 只 forEach 逐一呼叫。新增規則只要加一個 Validator。

第 16 頁 · Decorator

第九個 Decorator：想在嚴格策略上加「前後保留 30 分鐘緩衝」，直接改會讓邏輯複雜。改用 Decorator 包裝任意策略，在委派前後延伸時段，完全不改被包裝的策略，還能選擇性啟用。

第 17 頁 · Command

第十個 Command：Controller 直接呼叫 Service，稽核、非同步這些橫切關注點散在各處。我們把每個狀態操作封裝成 Command，由 CommandBus 統一執行並記稽核日誌。未來要加佇列、重試、非同步，都只改 CommandBus。

08:15–08:55

🔗 工作流程整合

第 18 頁 · 工作流程

這頁把完整流程依生命週期拆成四個階段，看模式如何協作：

階段一送出申請，走責任鏈三步驗證，再用 State 建立 Pending；階段二審核，用 Command 封裝、State 轉換、Observer 發 Email，主管還多一道同部門檢查；階段三、四的出車跟還車是同樣三模式協作，而還車若逾時，會自動建立違規記錄。

08:55–09:40

📐 收斂 ｜ 原則 + 測試 + 部署

第 19 頁 · OOD 原則

把模式背後的原則收斂一下：SOLID 五原則加上迪米特法則跟 DRY，都落在系統的具體位置——OCP 是三個模式的擴充免改、DIP 是 Repository 介面、DRY 是抽象權限守衛，並且全程組合優於繼承。

第 20 頁 · 測試策略

測試分兩層：單元測試用 InMemory Repository，不碰資料庫、毫秒級跑完，這正是依賴反轉帶來的可測試性；整合測試用 WebMvcTest 測 HTTP 層，涵蓋狀態碼、JWT 守衛，以及 401 跟 403。

第 21 頁 · CI/CD

部署是全自動的：推送 v 開頭的 tag 觸發 GitHub Actions，建置、上傳、PM2 重啟，最後經 Cloudflare Tunnel 對外。刻意只用 tag 觸發，精準控制部署時機。

09:40–10:00

🎓 結語

第 22 頁 · 結語

最後總結。我們實作了十個 GoF 模式，其中四個用於重構、消除痛點。後續還能擴充簡訊或 Line 通知、命令佇列等等。

三個收穫：第一，十個模式都對應真實痛點，不是為用而用；第二，OOD 原則貫穿分層架構；第三，介面與 DIP 讓系統可測試、可替換、可擴充。

以上是我們的報告，謝謝聆聽！
